# 跨 Spec 交叉审查

## 目的

当项目有多个功能 Spec 时，它们之间可能存在重合、冲突、依赖关系。跨 Spec 审查的目的是在开发之前就发现这些问题，避免写出不一致的代码。

即使只有一个 Spec，也应该检查其完整性。

## 审查流程

1. 读取 `docs/specs/` 下所有 `.md` 文件（排除 `*.test-plan.md`）
2. 对每个 Spec 检查完整性
3. 两两交叉比较检查重合、冲突、依赖
4. 生成报告，交互式引导用户处理

## 审查维度

### FAIL 级别（必须解决）

**规则冲突**：同一业务概念在不同 Spec 中定义不同

- 同一个状态/枚举在不同 Spec 中的值或术语不一致（比如两份 Spec 对"拒绝"用了"已拒绝"和"已驳回"两种措辞）
- 同一种操作在不同 Spec 中的业务规则矛盾
- 角色权限矛盾（A Spec 说某角色能做某事，B Spec 说不能）

**依赖缺失**：Spec A 提到了 Spec B 的功能，但 B 不存在

- 比如 Spec A 描述"扣减额度"，但没有对应的额度管理 Spec

### WARN 级别（建议解决）

**功能重合**：同一操作在多个 Spec 中都有描述

- 比如"审批流程"同时出现在多个业务 Spec 中
- 建议抽离为独立的通用 Spec 或在一个 Spec 中定义，其他引用

**循环依赖**：A 依赖 B，B 依赖 C，C 依赖 A

**完整性问题**：

- 某个 Section 为空或只有 TBD
- 有 F-NNN 功能项但没有对应的 BR-NNN 业务规则
- 有状态流转但某些状态没有描述如何到达或离开
- Edge Cases 少于 3 个

### INFO 级别（供参考）

**共享流程**：多个功能可能共享审批流、通知、余额计算等横切流程
**功能关联**：一个功能的操作可能触发另一个功能的更新（如一项申请通过 → 触发对应账户的额度变更）

## 报告格式

```
=== 跨 Spec 审查报告 ===
审查范围：[列出所有被审查的 Spec 文件名]
审查时间：[日期]

--- FAIL ---

[FAIL-001] spec-a.md × spec-b.md: "审批状态"定义不一致
  - spec-a.md BR-003: 状态为 待审批 → 已通过/已拒绝/已取消
  - spec-b.md BR-002: 状态为 待审批 → 已通过/已驳回
  → 建议：统一术语，"已拒绝"和"已驳回"应二选一

--- WARN ---

[WARN-001] spec-a.md 和 spec-c.md 都定义了独立的审批流程
  - spec-a.md Section 5: 提交 → 待审批 → 已通过/已拒绝
  - spec-c.md Section 5: 提交 → 待审批 → 通过/驳回
  → 建议：抽离通用 approval.md Spec，两者引用

[WARN-002] spec-d.md 完整性问题
  - Section 4（业务规则）有 3 个 TBD 标记
  → 请补充这些规则

--- INFO ---

[INFO-001] spec-a.md 和 spec-e.md 功能关联
  → 请确认：spec-a 的通过操作是否应触发 spec-e 记录的更新？
```

## 交互式处理

对每个 FAIL，逐一询问用户：

> "[FAIL-001] spec-a 和 spec-b 的审批状态术语不一致。你希望怎么处理？统一用'已拒绝'还是'已驳回'，还是有其他想法？"

对每个 WARN，逐一询问用户：

> "[WARN-001] spec-a 和 spec-c 都定义了独立的审批流程。要不要抽出一个通用的审批 Spec？还是保持独立？"

根据用户决定更新对应 Spec 文件。

## 更新规则

- 在受影响的 Spec 的 Section 8（跨功能关联）中记录审查结论
- 如果需要统一术语，更新所有相关 Spec 中的对应 Section
- 如果决定抽离通用 Spec，引导用户用 Phase 1 流程创建新 Spec
