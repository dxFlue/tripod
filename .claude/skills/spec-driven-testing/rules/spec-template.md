# Spec 格式定义与交互引导

## Spec 是什么

Spec 是功能需求文档，从用户和产品视角描述"这个功能/页面应该支持什么"。它不是技术文档——不涉及 API 端点、数据库表结构、DTO 定义等实现细节。

这样做的好处是：

- 项目还没有代码也能写 Spec（spec-first）
- 产品经理和开发者都能读懂
- 让测试用例设计聚焦在"功能应该怎么工作"而不是"代码应该怎么写"

## Spec 格式模板

```markdown
# [功能名称] 需求规格

## Meta

- **功能**: [名称]
- **版本**: 1.0
- **状态**: DRAFT | REVIEW | APPROVED
- **作者**: [user]
- **日期**: [ISO date]
- **关联功能**: [依赖或相关的其他功能，留空则写"无"]

## 1. 功能概述

[1-3 句话描述这个功能/页面是干什么的，解决什么问题]

## 2. 用户角色

谁会使用这个功能？每种角色分别能做什么？

| 角色 | 能做什么 | 不能做什么 |
| ---- | -------- | ---------- |

## 3. 功能清单

这个页面/功能应该支持哪些操作？

- F-001: [功能描述]
- F-002: [功能描述]

## 4. 业务规则

有哪些业务逻辑约束？

- BR-001: [规则描述]
- BR-002: [规则描述]

## 5. 流程与状态

如果功能涉及状态变化（如审批流、发货流），描述流转过程：

（无状态流转的功能可写"本功能不涉及状态流转"）

## 6. 界面交互预期

用户在界面上的操作流程和预期反馈。

## 7. Edge Cases

你能想到的异常/边界场景：

- EC-001: [描述]
- EC-002: [描述]

## 8. 跨功能关联

（此 Section 由 Phase 2 跨 Spec 审查时填充，初始留空）

- 依赖:
- 共享流程:
- 可能冲突:
```

## 交互式引导流程

一步一步引导用户填写 Spec，**每次只问一个问题**，等用户回答后再问下一个。不要一次性把所有问题抛出来。

### 第 1 步：功能名称

> "你要描述的功能/页面叫什么？给它起个名字。"

用用户给的名字作为 Spec 标题和文件名（转为 kebab-case 作为文件名）。

### 第 2 步：功能概述

> "用 1-3 句话描述它是干什么的、解决什么问题。不需要很正式，说清楚就行。"

### 第 3 步：用户角色

> "哪些角色会用到这个功能？每种角色分别能做什么、不能做什么？"

帮用户整理成表格格式。如果用户提到的角色不全面，可以追问："还有其他角色会接触到这个功能吗？比如管理员/审核者？"

### 第 4 步：功能清单

> "这个页面/功能应该支持哪些具体操作？请逐条列出来。"

帮用户编号为 F-001, F-002 ...。如果感觉列表不完整，可以根据已知角色追问。

### 第 5 步：业务规则

> "有哪些业务逻辑上的约束或规则？比如某些操作的前置条件、数量/金额限制、唯一性约束。"

帮用户编号为 BR-001, BR-002 ...。可以根据已列出的功能提示可能需要的规则。

### 第 6 步：流程与状态

> "有没有涉及状态流转的实体？比如一条记录从'草稿'到'已提交'再到'已完成'。如果有，请描述从开始到结束的完整流程。没有的话说'没有'就行。"

如果有，帮用户画出 ASCII 流转图。

### 第 7 步：界面交互

> "用户在界面上怎么操作？操作后看到什么反馈？哪些表单字段？哪些必填？"

### 第 8 步：Edge Cases

> "现在想想可能出错的场景——至少想 5 个。例如：输入为空时会怎样？两个人同时操作同一条记录会怎样？跨时区 / 跨日期边界的情况怎么处理？"

这一步非常重要，因为这里的每个 EC 都会直接变成 Track A 的测试用例。如果用户想不出 5 个，可以根据已有的功能清单和业务规则主动提示可能的边界场景（权限越界、并发、异常输入、网络失败、回滚）。

帮用户编号为 EC-001, EC-002 ...。

### 第 9 步：关联

> "这个功能和其他功能有什么依赖或共享的部分吗？"

## 完成后

将完整 Spec 呈现给用户审查，问："这份 Spec 有需要修改的地方吗？"

确认后保存到 `docs/specs/[module-name].md`。

提示用户下一步选项：

- `/spec-driven-testing review` — 审查所有 Spec 的一致性（推荐有 2 个以上 Spec 时执行）
- `/spec-driven-testing generate [name]` — 为这个功能生成测试计划
