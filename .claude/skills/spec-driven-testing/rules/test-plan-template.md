# 测试计划格式与三轨生成逻辑

## 测试计划输出格式

```markdown
# [功能名称] 测试计划

## Meta

- **来源 Spec**: [spec 文件名]
- **Spec 版本**: [版本号]
- **生成时间**: [ISO date]
- **三轨来源**: Track A (用户 edge case) + Track B (代码扫描) + Track C (Spec 推导)
- **三层输出**: Unit + API E2E (Playwright) + UI E2E (Playwright)

## 测试用例汇总

| Track              | 数量  | 描述                              |
| ------------------ | ----- | --------------------------------- |
| A                  | N     | 用户手动提供的 edge case          |
| B                  | N     | /graph-code-analysis 代码扫描发现 |
| C                  | N     | Spec 系统性推导                   |
| **合计（去重后）** | **N** |                                   |

### 按层级分布

| 层级 (Tier) | 数量 | 占比参考                                 |
| ----------- | ---- | ---------------------------------------- |
| Unit        | N    | 20-30%（纯函数/业务规则/边界值）         |
| API         | N    | 40-50%（权限矩阵/状态流转/并发）         |
| UI          | N    | 20-30%（核心用户流/表单交互/跨功能联动） |

## 测试用例

### 来源：Track A — 用户 Edge Case

| ID              | 需求引用 | 描述 | 优先级 | Tier        |
| --------------- | -------- | ---- | ------ | ----------- |
| TC-[PREFIX]-A01 | EC-001   | ...  | HIGH   | Unit/API/UI |

### 来源：Track B — 代码扫描

| ID              | 需求引用 | 描述 | 优先级 | Tier | 发现来源            |
| --------------- | -------- | ---- | ------ | ---- | ------------------- |
| TC-[PREFIX]-B01 | BR-003   | ...  | MEDIUM | API  | graph-code-analysis |

（如果 Track B 被跳过，写：Track B: Skipped — greenfield module, 无代码可扫描）

### 来源：Track C — Spec 推导

| ID              | 需求引用 | 描述 | 优先级 | Tier |
| --------------- | -------- | ---- | ------ | ---- |
| TC-[PREFIX]-C01 | F-001    | ...  | HIGH   | UI   |

### 最终合并（去重后）

| 序号 | ID             | 需求引用     | 描述 | 优先级 | Tier |
| ---- | -------------- | ------------ | ---- | ------ | ---- |
| 1    | TC-[PREFIX]-01 | F-001/BR-001 | ...  | HIGH   | UI   |
```

## Tier 分类决策规则

每个 TC 生成时必须标注 `Tier`，按以下优先级判断：

1. **Unit** —— 输入/输出是纯函数或内部逻辑，无需网络、无需数据库、无需浏览器
   - 日期/金额计算、单位换算、字符串格式化
   - 业务规则的边界判断（例如"余额 < 0 时拒绝"可以用一个 service 方法直接测）
   - 国际化 key 的翻译、schema 的 zod/pydantic 校验
   - **识别信号**：描述里只提"输入 X → 输出 Y"，没有提"用户点击/请求接口"

2. **API** —— 需要起服务端、鉴权、打接口才能验证
   - 权限矩阵：某角色调用某接口被拒
   - 状态流转：PENDING → APPROVED 的接口行为
   - 并发/事务：两个请求同时到达
   - 数据库持久化验证（打接口 + 查 DB）
   - **识别信号**：描述里有"HTTP/接口/响应码/JWT/token"等字眼

3. **UI** —— 必须通过浏览器交互才能覆盖
   - 表单验证在前端的即时反馈（必填/格式）
   - 组件状态：编辑模式切换、二次确认弹窗、toast 提示
   - 跨页面流程：列表 → 详情 → 编辑 → 返回列表
   - i18n 的前端渲染结果
   - **识别信号**：描述里有"点击/弹出/显示/跳转/加载态"等字眼

**冲突裁决**：同一个场景如果 Unit 能覆盖就优先选 Unit；如果 API 能覆盖且比 UI 便宜，就选 API；只有"非跑浏览器不可"的才放 UI。避免把能在低层级跑的用例塞到 UI 拖慢测试。

**可以在多个层级重复的场景**（不算重复，算不同视角）：

- 业务规则：Unit 测纯逻辑 + API 测接口拒绝 + UI 测用户看到的提示
- 但每层只测自己的视角，不要复制粘贴

## TC 编号规则

- **PREFIX**：从功能名取大写缩写（保持简短、清晰；如 `orders` → `ORD`，`user-profile` → `PROFILE`）
- **Track 标识**：A = 用户 edge case，B = 代码扫描，C = Spec 推导
- **分 Track 编号**：TC-PREFIX-A01, TC-PREFIX-B01, TC-PREFIX-C01 ...
- **合并后编号**：TC-PREFIX-01 ~ TC-PREFIX-NN（去掉 Track 标识，统一编号）

确认 PREFIX 时询问用户："这个功能的测试用例前缀用 `[PREFIX]` 可以吗？"

## Track A：用户 Edge Case 提取

直接从目标 Spec 的 Section 7（Edge Cases）中提取每个 EC-NNN，一一映射为测试用例。

映射规则：

- EC-001 → TC-PREFIX-A01
- 优先级默认 HIGH（用户能想到的往往是最关键的场景）
- 需求引用填对应的 EC-NNN

如果 Section 7 为空或不足 3 个 edge case，提醒用户补充。

## Track B：代码扫描（调用 `/graph-code-analysis`）

### 前置检查

根据 SKILL.md §0 识别出的项目目录约定搜索模块代码是否存在。典型位置（占位符，按项目替换）：

- `{backend_services_dir}/{module-name}/`
- `{frontend_pages_dir}/{module-name}/`
- 其他端（若有）的模块目录

如果以上路径都不存在 → 输出 "Track B: Skipped — greenfield module"，跳过。

### 执行

如果代码存在，告知用户：

> "检测到该功能已有代码实现，我将调用 /graph-code-analysis 对代码进行系统性分析，从中提取可能的测试用例。"

然后调用 `/graph-code-analysis [module-name]` 进行分析。

从分析结果中提取以下类型的发现，转化为测试用例：

- 缺失的错误处理路径 → 测试错误场景是否正确处理
- 权限校验缺失 → 测试未授权访问是否被拒绝
- 数据流断裂 → 测试数据从输入到输出的完整性
- 未覆盖的状态转换 → 测试非法状态变更是否被阻止
- 前后端契约不一致 → 测试实际行为是否与预期一致
- 边界条件 → 测试极值输入

## Track C：Spec 系统性推导

从 Spec 的 Section 3（功能清单）、Section 4（业务规则）、Section 5（流程与状态）、Section 6（界面交互）系统性展开。

### 展开维度

**1. 核心功能正常流（4-6 个）**
对每个 F-NNN，创建一个 happy path 测试：

- 创建操作 → 验证创建成功
- 查看列表 → 验证列表正常展示
- 查看详情 → 验证详情内容正确
- 编辑操作 → 验证编辑后数据正确
- 删除操作 → 验证删除成功

**2. 权限矩阵（8-12 个）**
对每个角色（Section 2）× 每个功能（Section 3），测试"不能做什么"：

- 非权限角色尝试执行受限操作 → 应被拒绝
- 普通用户尝试删除他人数据 → 应被拒绝
- 未登录用户访问受保护页面 → 应跳转登录

**3. 业务规则验证（6-10 个）**
对每个 BR-NNN，正反各一个测试：

- BR-001 正向：条件满足时操作成功
- BR-001 反向：条件不满足时被拒绝并给出提示

**4. 状态流转（4-8 个）**
从 Section 5 的状态图中提取：

- 每个合法转换：草稿 → 待审批 → 已通过 ✓
- 每个非法转换：已通过 → 待审批 ✗（应报错）
- 终态操作：已取消的记录不能再次审批

**5. 界面交互（3-4 个）**
从 Section 6 中提取：

- 列表筛选和排序是否正常
- 分页是否正常（空列表、大量数据）
- 表单必填项为空提交 → 应有提示

**6. 并发/竞态（1-2 个，如适用）**

- 两个管理员同时审批同一条记录
- 用户提交后立即取消

**7. 跨功能联动（2-4 个）**
从 Section 8（跨功能关联）中提取：

- 主操作通过 → 关联实体字段是否同步更新
- 主操作失败 → 关联副作用是否已回滚

**8. 边界值（4-6 个）**

- 空字符串/空值输入
- 极长文本输入
- 特殊字符输入
- 日期边界（跨月、跨年、周末、节假日）
- 数值边界（0、负数、最大值）

## 合并去重

为每个 TC 生成标准化签名：`[功能点]:[行为]:[预期结果]`

示例：

- `创建记录:前置条件不满足:拒绝并提示` — 如果 Track A 和 Track C 都有这个场景，保留描述更详细的

去重后重新编号为 TC-PREFIX-01 ~ TC-PREFIX-NN，保留原始 Track 来源标注供参考。

## 输出

保存到 `docs/specs/[module-name].test-plan.md`。

完成后提示用户：

- `/spec-driven-testing implement [name]` — 将测试计划转化为三层测试代码
- `/spec-driven-testing implement [name] unit|api|ui` — 只生成某一层
